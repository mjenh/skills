tapo — Architecture Discussion

Go client library for Tapo P100 smart plug control over LAN • v1 architecture walkthrough • July 2026

The Problem

Go developers building home or small-office automation need to control Tapo smart plugs locally. The existing Go library (rk295/tapo-go) is stale and lacks support for the KLAP protocol used by current firmware. Cloud control adds latency and privacy concerns. tapo fills this gap as a focused, zero-dependency, MIT-licensed module for P100 plugs.

Key Design Decisions

AD-1 Layered with internal/ boundary

All transport and crypto code is hidden behind Go's internal/ visibility fence. Consumers see only the root package: Plug, DeviceInfo, Option, and error sentinels. This means we can refactor protocol internals freely without breaking the public API.

Chosen
Layered + internal/ — clean public surface, internal testability via Transport interface
Rejected
Flat root package — simple but leaks transport details into go doc

AD-3 Thin Transport interface

The internal Transport interface has just two methods:

type Transport interface {
    Login(ctx context.Context, email, password string) error
    Send(ctx context.Context, method string, payload json.RawMessage) (json.RawMessage, error)
}

Plug builds the command JSON (e.g. {"method":"set_device_info","params":{"device_on":true}}), hands it to Send, and gets decrypted JSON back. The command vocabulary lives in one place — the root package — not duplicated across two transport implementations.

Chosen
Thin Send/Login — transport handles encryption only; interface never grows
Rejected
Command-level methods — GetDeviceInfo(), SetDeviceInfo() etc. duplicated in each transport

AD-7 Lazy negotiation via composite transport amended

NewPlug returns immediately — no network I/O. A NegotiatingTransport inside internal/transport implements the Transport interface and owns the protocol-selection logic: on the first Login call, it tries KLAP; on protocol failure, retries legacy; caches the winner for all subsequent calls.

Plug holds one Transport reference and never knows about KLAP or legacy directly. The WithTransport option bypasses negotiation by injecting a concrete transport.

Why amended: The adversarial reviewer found that without specifying who owns the negotiation loop, two implementers could build incompatible module structures. Placing it in a composite transport keeps Plug simple and the boundary clean.

AD-5 Mutex-based concurrency

Two mutex layers for goroutine safety:

Layer 1 (transport): Each transport impl guards its session state (cookies, keys, counters) with sync.Mutex.

Layer 2 (Plug): A separate mutex gates re-authentication. When concurrent goroutines hit session expiry, one re-auths while others block, then all retry with the fresh session.

Chosen
sync.Mutex — zero deps, sufficient for single-host client
Rejected
singleflight.Group — needs golang.org/x/sync; result-sharing not useful here (side-effect, not cache)

AD-6 Sentinel errors

Four sentinels cover the spec's required error categories:

SentinelWhen
ErrAuthInvalid credentials or login rejected
ErrTimeoutContext deadline or 10s default exceeded
ErrUnsupportedModelDevice is not P100 (warning, not failure)
ErrHandshakeProtocol handshake failed (both KLAP and legacy)

All wrapped via fmt.Errorf("...: %w", sentinel). Callers use errors.Is. Typed error structs deferred to a minor version if structured fields are needed.

Deferred Decisions

Per-command retry policy — Retry at Plug level or Transport level? Deferred until real-world usage reveals the right layer.
Structured logging — stdlib log/slog vs callback? A debug bool flag is sufficient for v1.
RSA-1024 key size — Tapo legacy uses RSA-1024, exactly at Go 1.24's floor. Monitor across Go releases.
Constant-time credential comparison — Revisit on security review.
